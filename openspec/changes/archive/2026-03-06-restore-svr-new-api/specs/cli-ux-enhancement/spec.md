## 修改需求

### 需求:美化输出样式

系统必须使用 `charmbracelet/lipgloss` 美化 CLI 输出，提供清晰易读的视觉体验。

#### 场景:成功消息输出

- **当** 生成成功完成
- **那么** 系统使用绿色样式输出成功标记（✓）
- **那么** 标题使用粗体样式
- **那么** 路径信息使用高亮样式

#### 场景:错误消息输出

- **当** 生成失败
- **那么** 系统使用红色样式输出错误标记（✗）
- **那么** 错误信息清晰可读
- **那么** 提供的建议使用不同的样式区分

#### 场景:进度信息输出

- **当** 处理多个服务时
- **那么** 系统以微服务为最小单位输出当前处理的服务名称
- **那么** 使用进度指示器（如 "[1/3] generating servora"）
- **那么** 最终输出总结信息（成功/失败统计）
- **那么** 失败明细逐条展示（服务名 + 错误信息）

### 需求:输出格式一致性

系统必须保持输出格式的一致性，参考 Ech0 项目的设计模式。

#### 场景:标题和消息格式

- **当** 输出任何信息
- **那么** 使用 "标题: 消息" 的格式
- **那么** 标题使用粗体和特定颜色
- **那么** 消息使用高亮样式

#### 场景:多行信息输出

- **当** 需要输出多行信息（如错误建议）
- **那么** 使用适当的缩进
- **那么** 保持视觉层次清晰

### 需求:交互式服务选择

系统必须使用 `charmbracelet/huh` 提供无参数时的交互式服务选择体验。

#### 场景:无参数触发交互

- **当** 用户执行 `svr gen gorm` 不带参数
- **那么** 系统显示交互式服务选择界面
- **那么** 用户可以通过方向键选择服务
- **那么** 支持多选模式

#### 场景:确认提示

- **当** 即将执行生成操作
- **那么** 系统显示确认提示
- **那么** 显示将要生成的服务列表
- **那么** 用户可以确认或取消操作

#### 场景:空选择退出

- **当** 用户在交互式服务选择中未选择任何服务
- **那么** 系统输出 "No services selected"
- **那么** 系统不执行任何生成操作
- **那么** 系统以退出码 0 结束

#### 场景:用户取消

- **当** 用户在确认阶段取消操作
- **那么** 系统输出 "Cancelled by user"
- **那么** 系统不执行任何生成操作
- **那么** 系统以退出码 0 结束

### 需求:颜色适配

系统必须支持浅色和深色终端主题。

#### 场景:自适应颜色

- **当** 在不同终端主题下运行
- **那么** 系统使用 `lipgloss.AdaptiveColor` 自动适配
- **那么** 浅色主题使用深色文字
- **那么** 深色主题使用浅色文字
- **那么** 保持可读性

### 需求:汇总输出与错误分类

系统必须在批量执行结束后输出可读且可解析的汇总结果，并对失败进行标准化分类。

#### 场景:汇总输出格式

- **当** 系统完成批量生成
- **那么** 第一行输出 `Summary: success=X failed=Y`
- **那么** 每条失败项使用 `- <service> [<error-type>] <message>` 格式输出

#### 场景:失败分类

- **当** 某服务执行失败
- **那么** 系统将错误分类为以下之一：`service-not-found`、`config-invalid`、`db-connect-failed`、`generation-failed`
- **那么** 汇总失败项中必须包含对应 `error-type`

#### 场景:退出码语义

- **当** 批量执行无任何失败
- **那么** 系统以退出码 0 结束
- **当** 批量执行存在任意失败
- **那么** 系统以退出码 1 结束

### 需求:CLI 必须暴露受支持的生成命令

`svr` CLI 必须将当前受支持的生成能力暴露为清晰的子命令集合，避免文档与帮助信息出现不一致。

#### 场景:根命令帮助展示生成命令
- **当** 用户执行 `svr --help`
- **那么** 帮助信息必须至少展示 `gen` 命令组
- **那么** 帮助信息必须展示 `new` 命令组

#### 场景:new 命令组展示 api 子命令
- **当** 用户执行 `svr new --help`
- **那么** 帮助信息必须展示 `api` 子命令
- **那么** `api` 子命令说明必须体现其服务维度输入方式
