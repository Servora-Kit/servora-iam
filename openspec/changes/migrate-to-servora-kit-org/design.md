## 上下文

项目当前使用 `github.com/Servora-Kit/servora` 作为 Go 模块路径和 proto 包的基础路径。随着项目从个人维护转向组织维护，需要将所有引用迁移到 `github.com/Servora-Kit/servora`。

当前状态：
- 约 125 个文件包含 `horonlee/servora` 引用
- 包括 proto 文件、Go 源代码、生成代码、配置文件和文档
- 所有服务（servora、sayhello）都使用相同的基础路径

约束：
- 必须保持代码生成流程正常工作（buf generate、wire、ent）
- 不能破坏现有的 DDD 分层结构
- 需要确保所有 import 路径一致性

## 目标 / 非目标

**目标：**
- 将所有 `github.com/Servora-Kit/servora` 引用更新为 `github.com/Servora-Kit/servora`
- 重新生成所有依赖路径的代码（proto、wire）
- 更新文档和配置中的 GitHub 链接
- 确保 `make gen`、`make api`、`make wire` 等命令正常工作

**非目标：**
- 不改变项目结构或架构
- 不修改业务逻辑
- 不更新外部依赖版本
- 不处理已发布版本的向后兼容（这是破坏性变更）

## 决策

### D1：使用全局查找替换而非手动更新

**选择：** 使用自动化脚本批量替换所有文件中的路径引用

**理由：**
- 125 个文件手动更新容易出错且耗时
- 路径替换是机械性操作，适合自动化
- 可以确保一致性，不会遗漏文件

**替代方案：**
- 手动逐文件更新：容易出错，效率低
- 只更新源文件，依赖重新生成：会遗漏文档和配置文件

### D2：先更新源文件，再重新生成代码

**选择：** 按以下顺序执行：
1. 更新所有 proto 文件的 go_package 选项
2. 更新所有 go.mod 文件
3. 更新所有手写 Go 源代码的 import
4. 更新配置和文档文件
5. 运行 `make api` 重新生成 proto 代码
6. 运行 `make wire` 重新生成依赖注入代码

**理由：**
- 生成代码依赖源文件，必须先更新源文件
- 分步执行便于验证每个阶段的正确性
- 如果生成失败，可以快速定位问题

**替代方案：**
- 同时更新所有文件：难以调试问题
- 先生成再更新：逻辑错误，生成会失败

### D3：保留归档文件中的旧路径

**选择：** 不更新 `openspec/changes/archive/` 中的历史文件

**理由：**
- 归档文件是历史记录，应保持原样
- 这些文件不参与当前构建流程
- 更新它们没有实际价值

**替代方案：**
- 更新所有文件包括归档：增加不必要的工作量

### D4：使用 sed 进行批量替换

**选择：** 使用 `find` + `sed` 组合进行批量文本替换

**理由：**
- sed 是标准 Unix 工具，可靠且高效
- 支持正则表达式，可以精确匹配
- 可以预览更改（-n 'p' 模式）

**替代方案：**
- 使用 Go 脚本：增加复杂度，不必要
- 手动编辑器查找替换：无法批量处理多文件

## 风险 / 权衡

**风险 1：生成代码可能包含意外的路径引用**
- 缓解：在重新生成前清理所有生成目录
- 缓解：生成后运行 `make test` 验证

**风险 2：某些文件可能被遗漏**
- 缓解：使用 grep 搜索确认所有引用都已更新
- 缓解：排除 .git 和 node_modules 等目录

**风险 3：破坏性变更影响外部用户**
- 权衡：这是必要的迁移，外部用户需要更新 import
- 缓解：在 README 中添加迁移说明

**风险 4：Wire 生成可能失败**
- 缓解：确保所有 import 路径更新后再运行 wire
- 缓解：如果失败，检查 wire.go 文件中的 import

## 迁移计划

1. **准备阶段**：
   - 备份当前状态（git commit）
   - 验证当前构建正常（make gen && make test）

2. **更新源文件**：
   - 更新所有 proto 文件的 go_package
   - 更新所有 go.mod 文件
   - 更新所有 Go 源代码的 import
   - 更新配置文件（buf.openapi.gen.yaml）
   - 更新文档文件（README.md、AGENTS.md 等）

3. **重新生成代码**：
   - 清理生成目录：`rm -rf api/gen/go`
   - 运行 `make api` 重新生成 proto 代码
   - 运行 `make wire` 重新生成依赖注入代码

4. **验证**：
   - 运行 `make test` 确保测试通过
   - 运行 `make lint.go` 确保代码规范
   - 手动检查关键文件的路径是否正确

5. **提交**：
   - 创建 git commit 记录变更
   - 更新 CHANGELOG（如果有）

**回滚策略**：
- 如果出现问题，使用 `git reset --hard` 回滚到备份点
- 重新运行 `make gen` 恢复生成代码
